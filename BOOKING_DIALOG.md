# Функционал диалога бронирования в Telegram боте

## Описание

Добавлен полный функционал для приема бронирований прямо из Telegram чата. Боту не нужна внешняя форма - пользователи могут забронировать стол прямо в приложении.

## Как это работает

### Для пользователя (гостя)

1. **Запуск диалога**: Пользователь пишет команду `/book` в чат с ботом
2. **Этапы диалога**:
   - 👤 **Имя** — бот просит указать имя (минимум 2 символа)
   - 📱 **Телефон** — просит телефон в формате +7XXXXXXXXXX или 8XXXXXXXXXX
   - 📅 **Дата** — просит дату бронирования (форматы: 25.03, 25.03.2026, 2026-03-25)
   - ⏰ **Время** — просит время в формате ЧЧ:МММ (например: 19:30)
   - 👥 **Количество гостей** — просит количество (от 1 до 20)
   - ✅ **Подтверждение** — показывает все данные и просит подтвердить

3. **Результат**:
   - ✅ При подтверждении бива сохраняется в БД
   - 🔔 Администраторы получают уведомление в админский чат
   - ✅ Пользователю отправляется подтверждение

### Для администраторов

- Брони из Telegram диалога появляются в том же формате, что и брони из веб-формы
- Админы видят карточку с полной информацией о брони
- Админы могут подтвердить, отменить, добавить теги (VIP, НЕ ЗВОНИТЬ и т.д.)
- Админы могут добавлять комментарии к гостю

## Технические детали

### Новые файлы

**`booking_dialog.py`** — модуль для управления диалогами бронирования
- Функции валидации (дата, время, телефон, количество гостей)
- Состояния диалога (FSM - Finite State Machine)
- Функции сохранения/получения состояния диалога из БД

### Обновленные файлы

**`tg_handlers.py`**:
- Добавлена обработка команды `/book`
- Добавлена логика для обработки текстовых сообщений в зависимости от состояния диалога
- Добавлена обработка callback queries для подтверждения/отмены бронирования
- Брони из диалога отправляются в админский чат (через `TG_CHAT_ID`)

### Используемые таблицы БД

Система переиспользует существующую таблицу `pending_replies` для хранения состояния диалога:
- `kind='booking_dialog'` — для отслеживания активного диалога пользователя
- Данные диалога сохраняются в JSON формате в поле `phone_e164`
- Время жизни состояния: 1 час

### Валидация входных данных

#### Телефон
- Принимает форматы: +7XXXXXXXXXX, 8XXXXXXXXXX, 7XXXXXXXXXX и т.д.
- Нормализует все в E.164 формат (+7XXXXXXXXXX)
- Проверяет что это валидный русский номер

#### Дата
- Принимает форматы: DD.MM, DD.MM.YYYY, YYYY-MM-DD
- Если не указан год, использует текущий год (или следующий если дата уже прошла)
- Не позволяет бронировать на дату в прошлом
- Максимум 365 дней в будущем

#### Время
- Принимает формат: ЧЧ:МММ
- Проверяет что ресторан работает (примерно 11:00 - 23:00)

#### Количество гостей
- Только целые числа от 1 до 20

⏰ В какое время?
Укажите время в формате ЧЧ:МММ (например: 19:30)

Укажите количество (от 1 до 20)

## Интеграция с существующей системой

- Новые брони сохраняются в таблицу `bookings` с `source='telegram_dialog'`
- Гости автоматически добавляются в таблицу `guests`
- События логируются в `booking_events` с `event_type='CREATED_FROM_BOT'`
- Админы получают карточку бронирования в своем чате (TG_CHAT_ID)
- Все остальные функции админа работают как обычно

## Конфигурация

Необходимые переменные в `config.py`:
- `TG_WEBHOOK_SECRET` — секретный ключ вебхука (уже использовается)
- `TG_CHAT_ID` — ID чата с админами (уже использовается)
- `PROMO_ADMIN_IDS` — ID администраторов (уже использовается)
